GA-Experiment1 Parameter and Metrics Reference
Source of truth: Experiments/Ex1-ShtPath-GA/GA-Experiment1.py
Generated: 2026-04-07

Purpose
- Simulation-based binary Genetic Algorithm (GA) for air-corridor edge orientation optimization.
- Decision variables are binary genes mapped to corridor edge groups:
  - 0 -> forward edges
  - 1 -> reverse edges
- Groups and variable count come from `identify_air_corridor_edge_groups` in Experiments/Ex1-ShtPath-GA/edge_grouping.py (imported by GA-Experiment1.py).

Core objective and validity rules
- Per-seed fitness contribution:
  fitness_seed = collisions + no_path_count + timeout_count
- Individual raw fitness over K seeds:
  fitness_raw = sum(collisions over seeds) + sum(no_path_count over seeds) + sum(timeout_count over seeds)
- Individual selection score:
  selection_score = fitness_raw / replications
- Planner-invalid pressure metric:
  planner_invalid_count = no_path_count + timeout_count
- Planner-invalid flag:
  is_planner_invalid = (planner_invalid_count >= invalid_threshold)
- Best individual seed dispersion metric:
  best_individual_seed_fitness_std = std(per_seed_fitness list)

GA operators and replacement
- Tournament selection uses selection scores (minimize); tournament size is `--tournament-size`. Two parents are drawn independently.
- Uniform crossover with probability `--crossover-prob`; if the uniform draw is >= that value, each child is a copy of its parent (no crossover).
- Bit-flip mutation per locus uses `--mutation-prob` (default `0.03`) for `mutation_prob`.
- Elitism: the `--elitism` individuals with lowest selection score are kept; offspring fill the remainder (population size minus elitism). Next population is elites stacked with offspring, trimmed to `--population`.
- If `--population` - `--elitism` <= 0 (including population == elitism), replacement is fully elitist: only the elite rows are carried forward (safe path for tiny diagnostic runs).

Early stopping (selection score)
- Improvement is judged on the generation best normalized selection score `best_individual_selection_score` (not raw fitness alone). The global tracker `best_global_selection_score` updates when that value improves.
- `no_improve_counter` resets to 0 on improvement; otherwise it increments each generation.
- Training stops when `no_improve_counter` reaches `--early-stop-patience` (consecutive generations without improvement on best selection score).

Terminal line interpretation (per generation)
- Printed metrics include:
  - best individual selection/raw/planner-invalid/seed-std
  - population selection mean/std
  - population invalid count ratio
  - population route-failure component means
  - population diversity
  - generation wall time
  - best seed scores as seed:score list

Replication / seed schedule (CRN protocol)
- Same seed list is used across all individuals inside a generation.
- All generations:
  - All individuals are evaluated with K=`--generation-seeds` (default 3).
  - No top-percentile refinement phase is applied.

Final-stage validation and sensitivity
- Candidate set: final population + global-best archive candidate.
- Ranking for shortlist: archived best-known normalized selection score.
- Final validation:
  - top `--final-validation-top-k` candidates (default 5)
  - held-out seed set K=`--final-validation-seeds` (default 20)
- Sensitivity:
  - enabled by default; disable with `--no-run-sensitivity`
  - one-bit flip sweep over best validated chromosome
  - evaluates delta fitness per bit on held-out seeds
  - bit sweep cap via `--sensitivity-max-bits` (0 means all bits)
- sensitivity_analysis.csv schema (written when sensitivity runs and at least one row is produced):
  - bit_index
  - base_bit
  - flipped_bit
  - base_fitness
  - flipped_fitness
  - delta_fitness
  - abs_delta_fitness
  - is_invalid_after_flip (same meaning as EvalResult.is_invalid / planner-invalid rule used elsewhere)

Cache and parallel evaluation behavior
- Cache key:
  (chromosome_bitstring, seed_signature)
- Duplicate chromosomes in same batch are de-duplicated.
- Parallel batch evaluation via thread pool with --workers.
